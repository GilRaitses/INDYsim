### Track Key Sorting - Root Cause Fix

**Problem Identified:**
- Track keys in H5 files were not stored in numeric order
- Analysis scripts processed tracks in H5 storage order, not track key order
- Monitor showed confusing display: "Processing track_9 (64/64)" when track_9 was 64th in storage order

**Solution Implemented:**

1. **Analysis Scripts (Defense in Depth):**
   - `scripts/queue/run_stimulus_locked_analysis_production.py`: Added track key sorting by numeric value before processing
   - `scripts/engineer_dataset_from_h5.py`: Added same sorting logic for consistency
   - Tracks now processed: track_1, track_2, ..., track_9, ..., track_64

2. **Conversion Scripts (Root Cause Fix):**
   - `src/@matlab_conversion/convert_matlab_to_h5.py`: Added explicit sorting `sorted(range(1, num_tracks + 1))` to ensure tracks stored in order
   - `scripts/2025-11-11/convert_matlab_to_h5.py`: Changed to two-pass approach (collect then sort) to ensure sorted storage
   - New H5 files will have tracks stored in numeric order from creation

**Result:**
- Sequential position (idx) now matches track key number
- Monitor shows: "Processing track_9 (9/64)" - track key matches position
- Tracks stored correctly from the start, not just sorted when reading
